The problem & my role

A "design system" that was just a sticker sheet.

When I joined, Screentone wasn't a system — it was a static file. Colors had creative but unscalable names like "Lava" and "Blurple." Components had no Auto-layout. Developers manually inspected hex codes and guessed padding values.

My mission: rebuild it from the ground up as a fluid, token-driven ecosystem aligned with Dow Jones's broader Index family — unlocking AI integration, zero-drift developer handoff, and effortless cross-brand scaling.

Problems

Four blockers preventing a scalable design ecosystem.

01

No shared language between design and code.

Without variables or tokens, developers had to visually inspect Figma files and manually extract hex codes and padding values. Every handoff created drift — the design and production UI were always slightly different, and CSS overrides multiplied with every sprint.

Add image: old Figma inspect panel / CSS drift example
02

Components that broke the moment you touched them.

With no Auto-layout, every size variant required detaching the component and manually regrouping elements. Designers spent hours on work that should have taken seconds — and each detach created a snowflake component that drifted from the library permanently.

Add image: old detached component / manual group structure
03

Color names that sounded good but scaled terribly.

"Lava", "Blurple", "Emerald" — evocative, creative, impossible to maintain. No programmatic relationship between values. When WCAG contrast requirements changed, there was no systematic way to audit or fix them. Every color decision was manual archaeology.

04

Screentone spoke a completely different language from Index.

Dow Jones's master ecosystem houses WSJ, MarketWatch, and other brands under a shared foundation called Index. Screentone was siloed — different naming conventions, different scale, different mental model. Cross-brand work required constant translation and produced inconsistent results.

Goal

One north star: make the system disappear.

A design system should be invisible infrastructure. Designers should never think about it — they just build. Developers should never guess — they just ship. The goal was to architect a system so coherent that the work it enables becomes effortless.

Research

Three key findings from auditing the system and talking to the team.

Before proposing anything, I spent two weeks doing a deep audit — reviewing every component, every token (or lack of one), and running structured interviews with designers and engineers who used the system daily.

01
"I just detach it — it's faster."

Every designer interviewed had developed a workaround for the same problem: components were too rigid to adapt without breaking. Detaching had become the default behaviour, not the last resort. The library was being abandoned in real time.

02
The dev handoff was a guessing game — not a spec.

Engineers described the Figma files as "a reference, not a source of truth." They'd build from the design, then adjust based on what felt right in the browser. Without semantic tokens, there was no contractual relationship between the design file and the codebase.

03
The 05–90 color scale was creating silent WCAG failures.

The irregular step scale meant contrast ratios between foreground and background colors weren't predictable. Developers were combining shades they believed would pass AA — and many didn't. A standardised 0–100 scale with predictable steps was the only way to make contrast auditable by design.

Design process

Five iterations to get from sticker sheet to living system.

01

Token audit — mapping what existed to what should exist

I catalogued every color, spacing value, and radius used across the product. Then mapped them against the Index token structure to identify gaps, redundancies, and the rename strategy needed to align with the broader DJ ecosystem.

Add image: token audit spreadsheet / mapping doc
02

4-tier token architecture — from raw hex to component contract

I designed a strict four-tier hierarchy: Raw Values → Design Tokens → Semantic Tokens → Component Tokens. Each tier has a single responsibility. Designers work with semantic tokens. Developers consume component tokens. The layers below are implementation details.

01
Raw Values
Hex codes and fixed px — #0057FF, 16px
02
Design Tokens
Universal language — blue-60, space-400
03
Semantic Tokens
Purpose-driven — color-error-text, color-primary-action
04
Component Tokens
Locked to UI — button-bg-primary, input-border-focus
03

0–100 color scale — accessibility baked in by default

Replaced the irregular 05–90 scale with a standardised 0–100 scale across all palettes. Even steps (20, 40, 60, 80) are light-on-dark safe; odd midpoints (30, 50, 70) are for decorative use only. WCAG AA contrast is now a structural guarantee, not a manual check.

Before — irregular scale
05 — 10 — 30 — 50 — 70 — 90
After — 0–100 standardised
0 — 20 — 40 — 60 — 80 — 100
04

Auto-layout rebuild — every component fluid by default

Rebuilt every component from scratch with strict Auto-layout and Figma variables wired to the token hierarchy. Properties replaced manual configuration. Designers can now drag, drop, and toggle — no detaching, no snowflakes, no drift.

05

AI-ready output — the token structure as a data contract

Because the system is built on a clean, predictable JSON token structure, it's now machine-readable. Claude can read our Figma specs. We can prompt AI to generate production-ready component code directly from token names — not from visual inspection.

"A well-structured token system is a well-structured data contract that any tool can consume."
